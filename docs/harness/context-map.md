# Context Map

The index of the harness. Read this to know which file holds which kind of
durable instruction, and which file to update when something changes.

## The five core layers

| Layer | Job | Primary files |
| --- | --- | --- |
| Context | Tell agents what matters and where to look | `AGENTS.md`, `context-map.md`, `constraints.md`, `project-context.md` |
| Tooling | Make checks executable instead of optional | `Makefile`, `scripts/harness.py` |
| Tooling | Guide structural code graph queries | `graph.md` |
| State | Make setup reproducible | `startup-readiness.md` |
| State | Preserve work across sessions | `progress.md`, `decisions.md` |
| Feedback | Turn failure into targeted harness improvements | `features.md`, `diagnostics.md`, `verification.md`, `quality.md`, `evaluator-rubric.md`, `sprint-contract-template.md` |

## Repository map

| `docs/harness/graph.md` | Code-review graph usage, query workflows, and install commands | Graph tooling, platforms, or usage rules change |
| `docs/harness/teaching-protocol.md` | Standing instruction: teach the human as work is done | The teaching method changes |

## Update protocol

When changing context:

1. Prefer the smallest file that holds your durable instruction.
2. Include source, applicability, and expiry for any durable instruction.
3. Keep `AGENTS.md` as a routing layer, not a binding document.
4. Add or update a verification command when a rule can be checked.
5. Run `make harness-check` when a rule is checked.
